Why a guard, not a type
distance_threshold is owned by the k-sweep loop — it sets n_clusters: k on every iteration, so the two parameters are mutually exclusive by construction. algorithm_params stays Record<string, unknown>: TypeScript cannot check 'distance_threshold' in an untyped spread at the type level, so a runtime guard at the caller boundary is the root-cause fix, not a type change.

What the guard lets through

The two boundary tests — as written

rejects distance_threshold at the boundary
// asserts the exact boundary message
const data = [[1,2],[2,3],[10,11],[11,12]];
await expect(
  find_optimal_clusters(data, {
    algorithm: 'agglomerative',
    algorithm_params: { distance_threshold: 2.5 },
  }),
).rejects.toThrow(
  "algorithm_params must not include 'distance_threshold'",
);
accepts linkage and metric via algorithm_params
// linkage + metric pass through and cluster
const { X } = make_blobs({
  n_samples: 30, n_features: 2, centers: 3,
  cluster_std: 0.3, random_state: 7,
});
const result = await find_optimal_clusters(X, {
  algorithm: 'agglomerative',
  algorithm_params: { linkage: 'complete', metric: 'euclidean' },
  min_clusters: 2, max_clusters: 4,
});
expect(result.optimal.k).toBeGreaterThanOrEqual(2);
expect(result.optimal.k).toBeLessThanOrEqual(4);
Deliberately left as-is
'distance_threshold' in is stricter than the downstream != null check — it rejects even { distance_threshold: undefined }, which matches the spec intent of forbidding the key outright. Two pre-existing issues stay out of scope: n_clusters inside algorithm_params still silently overrides the k-sweep, and AgglomerativeClustering keeps its own threshold validation — the boundary guard duplicates it on purpose, because AC#2 requires the error to surface at the find_optimal_clusters boundary.
