Drag n and the race re-runs — at small n the CPU wins, and the demo says so out loud

An n-slider (200 → 5000, hard cap) re-runs the race live, flipping the caption between "CPU wins" and "GPU pulls ahead" and marking the crossover n — a first-class small-n CPU win is what makes the demo un-riggable.

range 200 – 5000 hard cap n = 5000 re-run debounced, single-flight crossover fit a + b·n², solve builds on 55.4 race UI · 55.3 run_race
Comprehension job 1 · the crux · feel it before it is coded

Grab the slider — watch the winner, and the caption, flip at the crossover

What's really new here

55.4 already runs one race at the fixed default n=2000 and already shows a CPU win as "CPU is faster". 55.5 makes n a live input: a debounced, single-flight re-run, a flipping plain-language caption, and a computed crossover mark on the track.

Comprehension job 2 · the physics · why a crossover exists at all

Each lane's wall time is fixed overhead + compute — and the two have opposite shapes

Why the cap is n = 5000

The same n² grows the dense affinity matrix — 25M cells at n=5000 — which risks OOM/jank on mobile, so the slider hard-caps at 5000; compute runs in workers, so a large-n run never freezes the main thread (AC#4).

Comprehension job 3 · the detection · the mark is computed, never hard-coded

Fit each lane to t(n) = a + b·n² from observed runs, then solve for where they meet

Reading the formula

a_g − a_c is the GPU's extra fixed overhead (positive — the few-ms dispatch); b_c − b_g is CPU's steeper per-n² growth (positive). Their ratio is n*²; a non-positive or non-finite ratio means the curves never cross in range, so no mark is drawn.

Comprehension job 4 · the hazard · dragging fires a flood of input events

One drag = dozens of input events — debounce, then run latest-target-wins single-flight

Why this stays smooth (AC#4)

Every measured computation runs off-thread in the two workers (55.3, unchanged), so even n=5000 never blocks the main thread; the single-flight guard just prevents two worker-pairs at once, and debounce ties worker churn to landings, not raw events.
