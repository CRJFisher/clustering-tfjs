The crux · feel the Auto → override → parity-drop flip before reading it

Drag a control: the affected column re-clusters, and its honest badge is replaced by a neutral one

Why a single global slider would have been a lie

The 55.7 grid curates different params per cell: Spectral is nearest_neighbors on the curves but rbf on blobs; Agglomerative is single on thin manifolds but ward on blobs; n_clusters is 2 for moons/circles, 3 elsewhere. One flat global default would force a wrong param onto half the grid and break the parity story.

The one decision · how exploration is added without lying

Auto keeps the verified badge; the first explicit value overrides globally and drops the parity claim

Why this model and not the rejected alternative

Never advertising a parity you didn't verify is the grid's load-bearing AC. Auto preserves that contract and adds exploration; one flat global default would force a wrong param onto the curated rows and wear a badge it no longer deserves. Simpler, yes — but it would lie.

The surface · one live control per algorithm, each with a plain-English caption

Six controls, mapped to the real library parameter names, each scoped to the columns it touches

The pure core · curated params ⊕ overrides → effective params per cell

One unit-tested module turns a control into a per-cell effective-params set and the list of cells it touches

The wiring · the worker must outlive the initial sweep

The grid worker stays alive and caches the datasets, so a re-cluster is a lightweight message — not a fresh page load

The protocol gains a recluster verb — the rest is unchanged

grid_protocol.ts adds a recluster request (cell ids + effective params, no payloads) alongside the existing run; the worker keeps its cell_result / cell_error / progress outbound union exactly as 55.7 left it. The datasets are sent once with the first run and cached worker-side, so a re-cluster never re-ships the points.
