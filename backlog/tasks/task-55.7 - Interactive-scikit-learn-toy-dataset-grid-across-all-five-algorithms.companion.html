The crux · the grid as it now renders

The shipped grid — 25 real colored clusterings, one fit_predict per cell

The badge is data, not a sales claim

Every cell's tier comes from parity_for() in grid_config.ts and the footnote from count_parity(), so the "9 of 25 match exactly" line can never drift from the badges. Where float32 moves a few boundary points, the cell is marked ≈, never re-tuned to fake a match.

The wiring · why the page never freezes

Main thread generates the datasets; one worker clusters; each cell renders once

One sequential worker, not a pool

The toy data is tiny (300 points per dataset) and the tfjs engine stays warm across the 25 fits, so a single sequential worker beats a pool — which would re-pay backend init for data that fits in milliseconds.

The parity story · read it by scanning a column

The four honest tiers — computed from config, never hand-typed

The open delivery question is resolved

The exact 25-cell parameter table — n_clusters, Spectral's nearest_neighbors vs rbf affinity, Agglomerative's single vs ward linkage, per-row min_cluster_size, and the SOM grid + cluster(n) linkage — lives in grid_config.ts behind a fully-typed discriminated union (no casts). One curated set serves every row because every dataset is standardized to zero-mean / unit-variance first.

The one real integration bug · found and fixed

tfjs chained ops get tree-shaken out of the worker bundle

Why the race worker was immune

The race worker only awaits a functional affinity matrix, never a chained op, so its registration survived tree-shaking. The grid worker runs full fit_predict across all five estimators, every one of which leans on chained tensor methods — so only it needs the explicit import.
