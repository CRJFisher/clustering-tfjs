how a reference number reaches an assertion

The HDBSCAN fixture pipeline

A scikit-learn run produces a JSON fixture carrying both the final labels/probabilities and sklearn's own single-linkage hierarchy. That dual payload is what lets the same fixture feed two assertions of very different strictness — one bit-exact, one tolerance-based — depending on which stage of the pipeline is under test.

the subtle part

Why some assertions are exact and others cannot be

The divergence is born in exactly one place: when two MST edges carry equal mutual-reachability weight, their merge order is undetermined. scikit-learn (via numpy's unstable argsort) and this implementation (Prim's, by insertion order) may break that tie differently. Downstream, a different merge order can flip which boundary point falls out of which cluster first — shifting a handful of labels and their probabilities. The condensed-tree core is innocent: given the same hierarchy, it is deterministic and exact.

Path A — exact

Input is sklearn's hierarchy, so the only ambiguous step (edge sort) has already been resolved by sklearn. Both sides condense the identical merge order.

condensation_tree.test.ts:88–107
const tree = condense_hierarchy(
  fixture.single_linkage_tree, n, mcs);
const { labels, probabilities } =
  extract_labels(tree, selected, n);
// exact, up to cluster-id permutation
expect(labels_equivalent_with_noise(
  labels, fixture.labels)).toBe(true);
expect(probabilities[i])
  .toBeCloseTo(fixture.probabilities[i], 6);

Path B — tolerance

Input is raw X. The implementation builds its own MST; tied weights may sort differently than numpy did, so a few boundary points diverge.

hdbscan.test.ts:84–103
const labels = await model.fit_predict(data);
// MST tie-ordering ⇒ cannot be exact
expect(mine_clusters).toBe(sk_clusters);
expect(alignment_agreement(labels,
  fixture.labels)).toBeGreaterThanOrEqual(0.95);
// blanket bound today — to be tightened:
expect(mae / probs.length)
  .toBeLessThanOrEqual(0.2);

tightening the loosest assertion

From blanket MAE to documented bound

Today every fixture passes under one blanket rule: mean absolute probability error ≤ 0.2. That is far looser than most fixtures need. The tightening classifies each fixture by whether its mutual-reachability weights contain ties, then asserts as strictly as the fixture allows.

library-defined, no sklearn equivalent

Exemplars and medoids: behavioural-only coverage

exemplar_indices_ (HDBSCAN, the most-persistent point per cluster) and medoid_indices_ (the real sample nearest each cluster mean) have no scikit-learn counterpart to diff against. They are documented in code as library-defined and pinned by behaviour instead of by reference output.

documentation deliverable

Both properties get a code comment stating explicitly: library-defined, no scikit-learn equivalent; correctness is defined by the behavioural contract above, not by reference parity. This closes the acceptance criterion and prevents a future reader from mistaking the tolerance for an unfinished sklearn comparison.
