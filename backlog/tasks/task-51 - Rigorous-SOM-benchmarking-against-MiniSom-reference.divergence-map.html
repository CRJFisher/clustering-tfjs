The shiftFrom loose tolerances to per-element parity

The reference suite compares a deterministic MiniSom replica against fixtures generated by MiniSom itself. Earlier the suite could only assert that the two trainers ended up roughly in the same place, because the production online mini-batch trainer disagrees with MiniSom on nearly every step of the weight update.

The suite now asserts exactness. A separate, deterministic reference path replicates MiniSom step for step, identical initial weights are injected, and weights, quantization error and the U-matrix are checked element by element at 1e-9 (the observed gap is around 1e-12), with BMUs and labels matched exactly. The production online mini-batch path is unchanged — the rigor lives entirely in the reference trainer.

Why topographic error is the one ≤1% metric, not 1e-9 Topographic error is a discrete measure: it counts samples whose first- and second-nearest neurons are non-adjacent, and the second-nearest selection is discontinuous at distance degeneracies (two neurons exactly tied). A weight difference of 1e-12 can flip which neuron wins a tie, moving the count by one sample. So TE is held to a ≤ 1% relative bound rather than an element tolerance — and across the 32 fixtures, 31 land at 0% deviation and one at 0.24%, well inside the bound.

The premise that drove the designWhat MiniSom's train_batch really does

A "batch-map" SOM (w ← Σ(h·x)/Σ(h) over the whole dataset per epoch) is the intuitive thing to replicate — but MiniSom's train_batch is not a batch-SOM. It is fully online, per-sample, and sequential, and entirely deterministic given fixed initial weights. The reference trainer replicates exactly this.

minisom.py — what train_batch actually runs
def train_batch(self, data, num_iteration, verbose=False):
    # NOT a batch SOM — just train() with sequential ordering
    self.train(data, num_iteration, random_order=False, verbose=verbose)

def train(self, data, num_iteration, random_order=False, ...):
    for t in range(num_iteration):
        idx = t % len(data)            # deterministic sequential cycling — NO RNG
        x = data[idx]
        win = self.winner(x)               # BMU for THIS single sample
        self.update(x, win, t, num_iteration) # whole-grid update, asymptotic decay at t

def update(self, x, win, t, max_iteration):
    eta   = self._decay_function(self._learning_rate, t, max_iteration)
    sig   = self._decay_function(self._sigma,         t, max_iteration)
    g     = self._neighborhood(win, sig) * eta
    self._weights += einsum('ij,ijk->ijk', g, x - self._weights) # raw update, NOT normalized
The load-bearing fact, confirmed in the build The only RNG in the whole run is random_weights_init (it picks random data rows as the starting grid). With identical initial weights injected, train_batch is a pure deterministic function — there is no RNG in the order. The reference trainer in som_reference_training.ts reproduces this loop tensor-free, and the resulting weights match MiniSom to ~1e-12.

Update & decayThe formulas that line up

Two pairs of formulas decide parity. The reference trainer adopts the MiniSom side of each exactly; production keeps the left side untouched.

Weight update

Production — normalized mini-batch

wⱼ += lr · Σᵢ(hᵢⱼ·(xᵢ−wⱼ)) / Σᵢ(hᵢⱼ)

Sum over a 32-sample mini-batch, divided by the summed influence so the step is batch-size independent. Order-dependent because the batch is shuffled. Left untouched by this work.

Reference — raw per-sample (MiniSom)

w += η(t) · g(t) · (x − w)

One sample x, whole-grid update, g centred on its BMU. No normalization by Σ influence. Applied num_iteration times over the sequentially-cycled dataset. This is what som_reference_training.ts computes.

Decay schedule (learning rate and sigma share the same form)

Production — exponential, per epoch

p(ep) = final + (init−final)·e−ep / (E/5)

Stepped once per full pass; floor at a final value. Untouched.

Reference — asymptotic, per iteration (MiniSom)

p(t) = p₀ / (1 + t / (T/2))

Stepped every single sample; T = num_iteration. Applied identically to learning rate and sigma. The reference trainer reproduces this per-iteration lr/sigma sequence exactly, which is the foundation of the 1e-12 weight agreement.

The subtle traps, closedNeighborhood functions & hex coordinates

Matching the update loop is not enough — the neighborhood g and the grid coordinates feeding it must match too. Three of these are genuine formula differences, not just constants; all are now replicated and verified against MiniSom.

gaussian — separable product verified

MiniSom: e−Δx²/2σ² · e−Δy²/2σ²
ours: e−(Δx²+Δy²)/2σ²

The separable product equals the euclidean form on a rectangular grid; on hexagonal it differs only through the coordinates. The reference path uses MiniSom's separable form over its _xx,_yy directly.

bubble — box, not disc verified

MiniSom: (cx−σ < x < cx+σ) ⊗ (cy−σ < y < cy+σ)
ours: d ≤ σ (disc)

A square open-interval box with strict <, not a disc. The reference path implements the open-interval box exactly; production keeps its disc.

mexican_hat — MiniSom wavelet verified

MiniSom: e−p/d·(1 − 2p/d),  p=Δ², d=2σ²
ours: 2·(1−(d/σ)²)·e−(d/σ)²/2

A different formula and amplitude. The reference path implements MiniSom's exact form, and mexican_hat fixtures across square and non-square grids match to 1e-12.

hex coordinates — offset parity verified

MiniSom: _xx[::-2] −= 0.5 ; _yy *= 0.8660254
ours: x = col + (row%2)·0.5 ; y = row·√3/2

The √3/2 ≈ 0.866 factor matches; MiniSom offsets every other row counting from the LAST row, so parity depends on grid height. The reference path follows MiniSom's convention exactly, and the 8×4/4×8 non-square hexagonal fixtures confirm both the offset parity and the axis transpose bit-for-bit.

Coverage that exposes these traps The fixture set now includes non-square grids (8×4 and 4×8) and mexican_hat configurations across both rectangular and hexagonal topologies. These were the cases that previously could pass silently with a transposed axis order or a flipped hex offset parity — they now actively verify both.

The decision madeApproach A — replicate MiniSom's real train_batch

Two strategies were on the table. The decision resolved to Approach A: replicate MiniSom's real train_batch in a deterministic reference path. A was built; B was not.

A · BUILT

Replicate MiniSom's real train_batch

A reference-only training path that runs the deterministic online-sequential loop: idx = t % n, raw per-sample whole-grid update, asymptotic per-iteration decay, separable neighborhoods, MiniSom hex coords — with injected identical initial weights. This is the path that shipped, as som_reference_training.ts.

  • Validates directly against the actual MiniSom output
  • Deterministic given fixed init → weights/QE/U-matrix match to ~1e-12
  • Exact BMUs and labels; topographic error inside ≤ 1%
  • Production online mini-batch path untouched
  • Requires a faithful port of decay + neighborhood + hex coord details — done and verified
B · NOT BUILT

Implement a true batch-map SOM

The intuitive alternative: accumulate w ← Σ(h·x)/Σ(h) over the whole dataset per epoch in both Python and TypeScript. Rejected at decision time.

  • MiniSom does not compute a batch-map — train_batch is online
  • Would not validate against the MiniSom reference at all
  • Contradicts the task's own intent ("match MiniSom")
  • Doubles the implementation surface for no parity gain
  • (Only upside) batch-map converges smoothly — but irrelevant here
Decision — resolved

Approach A was taken and built. Because train_batch is deterministic once initial weights are fixed, the faithful replica reaches per-element parity around 1e-12 (asserted at 1e-9) on weights, quantization error and the U-matrix, with exact BMUs and labels and topographic error inside the ≤ 1% bound. Approach B would have computed something MiniSom never computes, so it could not validate against the fixtures — it was not built.

Where the reference trainer livesA dedicated, tensor-free module

The MiniSom replica exists only to validate the production trainer, so its math is quarantined out of the production API. It lives in a dedicated module, src/clustering/som_reference_training.ts, computed tensor-free with plain arrays, and is consumed only by the reference test. The production SOM class keeps its clean online mini-batch surface.

One piece does belong on production: initial_weights. It is a genuine reproducibility feature any caller benefits from, so it is a real SOMParams field, honored by both fit and partial_fit. Injecting it is what makes train_batch deterministic and the 1e-12 parity reachable.

As-builtWhat was built, file by file

The work across seven surfaces. Production training behavior is unchanged apart from the initial_weights feature; the rigor lives in the reference trainer, the regenerated fixtures, and the docs.

1

src/clustering/som.ts — initial_weights production feature

Accepts a [grid_height][grid_width][n_features] initial weight grid as a real SOMParams field, used verbatim instead of RNG init. Honored by both fit and partial_fit for reproducibility. The online mini-batch update and decay are unchanged.

2

src/clustering/som_reference_training.ts — reference trainer

A dedicated, tensor-free module replicating MiniSom's real train_batch: sequential idx = t % n, raw per-sample whole-grid update, asymptotic per-iteration decay p₀/(1+t/(T/2)), separable gaussian / open-box bubble / MiniSom mexican_hat, and MiniSom hex coords (offset-from-last-row, ·0.8660254). Computed with plain arrays and consumed only by the reference test.

3

src/clustering/som_neighborhood.ts — parity helpers

Provides the MiniSom-equivalent neighborhood and grid-coordinate variants the reference trainer needs (separable gaussian, open-interval box bubble, MiniSom mexican_hat, hex coords with correct offset parity and axis order). Production helpers are unchanged.

4

tools/sklearn_fixtures/generate_som.py — rewritten generator

Rewritten on a snake_case schema with num_iteration as the per-sample step count. Injects and exports initial_weights (the exact grid seeded into MiniSom) and records MiniSom-native metrics. Uses the real train_batch. Scenarios expanded to 32: 4 datasets × {square + 8×4 / 4×8 non-square} × {gaussian, bubble, mexican_hat} × {rectangular, hexagonal}.

5

__fixtures__/som/*.json — 32 regenerated fixtures

Each fixture stores X, params, initial_weights, final weights, bmus, labels, quantization_error, topographic_error, u_matrix — all snake_case, with init weights present.

6

src/clustering/som_reference.test.ts — tight assertions, no skips

Injects initial_weights, runs the reference trainer, and asserts weights / QE / U-matrix per element at 1e-9, exact BMU/label match, and topographic error within ≤ 1% relative. No it.skip remains. The online-mode property tests in som.test.ts / som_hexagonal.test.ts stay intact and clearly separated.

7

docs/som-benchmarking.md — canonical benchmarking guide

Documents the MiniSom venv setup, the regeneration command, what each fixture field means, the asymptotic decay form, the deterministic-equivalence guarantee, and the tolerance rationale. Regeneration needs the MiniSom venv and is not run in CI; the reference Jest suite is.

Reference: the exact parameters the reference trainer is fed

For each fixture, the reference test configures the trainer with the fixture's initial_weights, num_iteration read as MiniSom per-sample steps (not full passes), the asymptotic decay form for both learning rate and sigma, the topology-appropriate neighborhood, and MiniSom's hex coordinate convention. The earlier generator's conflation of per-sample iterations with epochs is gone: the schema and the reference path both treat the count as per-sample iterations.
