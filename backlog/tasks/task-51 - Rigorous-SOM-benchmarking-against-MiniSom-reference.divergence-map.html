Task-51 · SOM reference benchmarking

Matching MiniSom exactly: the divergence map and the equivalence strategy

The SOM reference fixtures pass only under loose tolerances because the production trainer and MiniSom differ in every moving part of the update. This lays out exactly where they diverge, what MiniSom's train_batch truly computes, and the one strategy that reaches numeric parity — which is the opposite of the plan's first instinct.

The problemWhy the current tolerances are loose

The reference suite compares the production SOM.fit() against fixtures generated by MiniSom. Because the two trainers disagree on nearly every step of the weight update, the suite can only assert that they end up roughly in the same place: average weight difference under 2.0, quantization error within 60% relative, label agreement above 0.5, with blobs_10x10_gaussian_rectangular skipped entirely.

Those bounds catch shape errors and gross regressions but not subtle algorithmic correctness. The fix is not to nudge the production trainer toward MiniSom — it is to add a separate, deterministic reference path that replicates MiniSom exactly, inject identical initial weights, and assert tight tolerances against it. The production online path stays exactly as it is.

Correcting the plan's premiseWhat MiniSom's train_batch really does

The Implementation Plan recommends adding a "batch-map" mode (w ← Σ(h·x)/Σ(h) over the whole dataset per epoch) and assumes MiniSom's train_batch is a batch-SOM. It is not. train_batch is fully online, per-sample, and sequential — and entirely deterministic given fixed initial weights.

minisom.py — what train_batch actually runs
def train_batch(self, data, num_iteration, verbose=False):
    # NOT a batch SOM — just train() with sequential ordering
    self.train(data, num_iteration, random_order=False, verbose=verbose)

def train(self, data, num_iteration, random_order=False, ...):
    for t in range(num_iteration):
        idx = t % len(data)            # deterministic sequential cycling — NO RNG
        x = data[idx]
        win = self.winner(x)               # BMU for THIS single sample
        self.update(x, win, t, num_iteration) # whole-grid update, asymptotic decay at t

def update(self, x, win, t, max_iteration):
    eta   = self._decay_function(self._learning_rate, t, max_iteration)
    sig   = self._decay_function(self._sigma,         t, max_iteration)
    g     = self._neighborhood(win, sig) * eta
    self._weights += einsum('ij,ijk->ijk', g, x - self._weights) # raw update, NOT normalized
The single load-bearing correction The only RNG in the whole run is random_weights_init (it picks random data rows as the starting grid). Inject identical initial weights and train_batch becomes a pure deterministic function. The plan's stated reason for rejecting "port MiniSom's online order" — "brittle, RNG order" — is unfounded: there is no RNG in the order.

Update & decayThe formulas that must line up

Two pairs of formulas decide parity. The reference path adopts the MiniSom side of each; production keeps the left side untouched.

Weight update

Production — normalized mini-batch

wⱼ += lr · Σᵢ(hᵢⱼ·(xᵢ−wⱼ)) / Σᵢ(hᵢⱼ)

Sum over a 32-sample mini-batch, divided by the summed influence so the step is batch-size independent. Order-dependent because the batch is shuffled.

MiniSom — raw per-sample

w += η(t) · g(t) · (x − w)

One sample x, whole-grid update, g centred on its BMU. No normalization by Σ influence. Applied num_iteration times over the sequentially-cycled dataset.

Decay schedule (learning rate and sigma share the same form)

Production — exponential, per epoch

p(ep) = final + (init−final)·e−ep / (E/5)

Stepped once per full pass; floor at a final value.

MiniSom — asymptotic, per iteration

p(t) = p₀ / (1 + t / (T/2))

Stepped every single sample; T = num_iteration. Applied identically to learning rate and sigma. This is the schedule a reference path must replicate, and the per-iteration lr/sigma sequence is the first thing to verify.

The subtle trapsNeighborhood functions & hex coordinates

Matching the update loop is not enough — the neighborhood g and the grid coordinates feeding it must match too. Three of these are genuine formula differences, not just constants.

gaussian — equal on rectangular

MiniSom: e−Δx²/2σ² · e−Δy²/2σ²
ours: e−(Δx²+Δy²)/2σ²

The separable product equals our euclidean form on a rectangular grid. On hexagonal it differs only because _xx,_yy differ (see coords below).

bubble — box vs disc differs

MiniSom: (cx−σ < x < cx+σ) ⊗ (cy−σ < y < cy+σ)
ours: d ≤ σ (disc)

A square open-interval box, not a disc — and strict < vs our ≤. Both must change in the reference path.

mexican_hat — different wavelet differs

MiniSom: e−p/d·(1 − 2p/d),  p=Δ², d=2σ²
ours: 2·(1−(d/σ)²)·e−(d/σ)²/2

Different formula and amplitude. New mexican_hat coverage is an explicit acceptance criterion, so the reference path must implement MiniSom's exact form.

hex coordinates — offset parity risk

MiniSom: _xx[::-2] −= 0.5 ; _yy *= 0.8660254
ours: x = col + (row%2)·0.5 ; y = row·√3/2

The √3/2 ≈ 0.866 factor matches. But MiniSom offsets every other row counting from the LAST row, so offset parity depends on grid height — ours always offsets odd rows. All current fixtures are square (5×5, 6×6, 7×7, 10×10), which also hides the axis-transpose; a non-square fixture is needed to expose both.

Coverage gap to close Add at least one non-square grid and at least one mexican_hat fixture. Square-only coverage silently passes even with a transposed axis order or a flipped hex offset parity.

The heart of the matterWhich equivalence strategy

The plan and reality point in opposite directions. The genuinely correct strategy is to replicate MiniSom's real train_batch — not a batch-map.

A · RECOMMENDED

Replicate MiniSom's real train_batch

A reference-only training path that runs the deterministic online-sequential loop: idx = t % n, raw per-sample whole-grid update, asymptotic per-iteration decay, separable neighborhoods, MiniSom hex coords — with injected identical initial weights.

  • Validates directly against the actual MiniSom output
  • Deterministic given fixed init → reaches ~1e-12 weight parity
  • Far exceeds every acceptance criterion (1e-3, 1%, exact)
  • Production online mini-batch path untouched
  • Must faithfully port decay + neighborhood + hex coord details
B · REJECTED

Implement a true batch-map SOM

The plan's instinct: accumulate w ← Σ(h·x)/Σ(h) over the whole dataset per epoch in both Python and TypeScript.

  • MiniSom does not compute a batch-map — train_batch is online
  • Would not validate against the MiniSom reference at all
  • Contradicts the task's own intent ("match MiniSom")
  • Doubles the implementation surface for no parity gain
  • (Only upside) batch-map converges smoothly — but irrelevant here
Decision

Take Approach A. Because train_batch is deterministic once initial weights are fixed, a faithful replica reaches per-element parity around 1e-12 — three orders of magnitude tighter than the 1e-3 criterion, with exact BMUs and labels and QE/TE far inside 1%. The plan's batch-map (B) would compute something MiniSom never computes, so it could not validate against the fixtures at all.

Secondary design questionWhere the reference trainer should live

The task's Key files list implies the reference loop lands on the production SOM class in som.ts. The constitution forbids surplus code in production, which creates a tension: the MiniSom replica exists only to satisfy tests, yet a reference mode on SOM would expose MiniSom-specific math through the production API.

a · reference mode on SOM

A flag/method on the production class

  • Matches the task's stated Key files
  • Leaks MiniSom-replica math into the production surface
  • Tension with "no surplus code in production"
b · RECOMMENDED — dedicated module

Isolated reference-trainer module

  • MiniSom-replica math clearly quarantined, used only by the reference test
  • Production SOM API stays clean
  • initial_weights injection stays a genuine production feature (reproducibility)
  • One more file to maintain alongside som.ts
Recommendation Keep initial_weights injection on production SOM — it is a legitimate reproducibility feature any caller benefits from. Put the MiniSom-equivalent loop in a clearly-labelled reference module (e.g. som_minisom_reference.ts) consumed only by som_reference.test.ts, so the production training path keeps its clean online mini-batch API.

HandoffWhat changes, file by file

The shape of the work across six surfaces. Production training behavior does not change; the rigor lives in the reference path, the regenerated fixtures, and the docs.

1

src/clustering/som.ts — add initial_weights injection

Accept a [grid_height][grid_width][n_features] initial weight grid and use it verbatim instead of RNG init. This is a real production feature (reproducibility), so it stays on the class. No change to the online mini-batch update or decay.

2

src/clustering/som_minisom_reference.ts — new reference trainer

A quarantined module replicating MiniSom's real train_batch: sequential idx = t % n, raw per-sample whole-grid update, asymptotic per-iteration decay p₀/(1+t/(T/2)), separable gaussian / open-box bubble / MiniSom mexican_hat, and MiniSom hex coords (offset-from-last-row, ·0.8660254). Used only by the reference test.

3

src/clustering/som_neighborhood.ts — parity helpers

Provide the MiniSom-equivalent neighborhood and grid-coordinate variants the reference trainer needs (separable gaussian, open-interval box bubble, MiniSom mexican_hat, hex coords with correct offset parity and axis order). Production helpers stay as they are.

4

tools/sklearn_fixtures/generate_som.py — rewrite the generator

Export initial_weights (the exact grid seeded into MiniSom). Fix the key casing — it currently writes camelCase (gridWidth, numEpochs, uMatrix) and stores num_iteration as numEpochs, while the on-disk fixtures are snake_case with no init weights, so the generator does not match its own output. Keep using the real train_batch. Expand scenarios: add a non-square grid and a mexican_hat config.

5

__fixtures__/som/*.json — regenerate

Each fixture stores X, params, initial_weights, final weights, bmus, labels, quantization_error, topographic_error, u_matrix — all snake_case, with init weights now present.

6

src/clustering/som_reference.test.ts — tighten, no skips

Inject initial_weights, run the reference trainer, and assert weights ≤ 1e-3 per element, QE/TE ≤ 1% relative, exact BMU/label match. Remove the it.skip on blobs_10x10_gaussian_rectangular. Leave the online-mode property tests in som.test.ts / som_hexagonal.test.ts intact and clearly separated.

7

docs/ or backlog/docs/ — SOM benchmarking guide

Document the venv setup, regeneration command, what each fixture field means, the asymptotic decay form, the deterministic-equivalence guarantee, and the tolerance rationale. Note that regeneration needs the MiniSom venv and is not run in CI; the reference Jest suite is.

Reference: the exact reference-mode parameters to feed

For each fixture, the reference test should configure the trainer with the fixture's initial_weights, num_epochs read as MiniSom num_iteration (per-sample steps, not full passes), the asymptotic decay form for both learning rate and sigma, the topology-appropriate neighborhood, and MiniSom's hex coordinate convention. The current generator's num_iteration → numEpochs conflation is one of the bugs being fixed: the reference path must treat that count as per-sample iterations.
