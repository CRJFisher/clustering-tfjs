Model selection · find_optimal_clusters

A loop-owned parameter must be rejected at the door, not deep inside the clusterer

algorithm agglomerative guard at input validation no type change +1 reject test · +1 accept test
Source · what produces the collision
find_optimal_clusters.ts:276 spreads { n_clusters: k, ...algorithm_params } into new AgglomerativeClustering(...); agglomerative.ts:282 throws when both are set.

Where the error surfaces — current vs planned

Why the guard, not a type
distance_threshold is owned by the k-sweep loop — it sets n_clusters: k on every iteration, so the two parameters are mutually exclusive by construction. algorithm_params stays Record<string, unknown>: TypeScript cannot check 'distance_threshold' in an untyped spread at the type level, so a runtime guard at the caller boundary is the root-cause fix, not a type change.

The change — one guard at input validation

What the guard lets through

The two boundary tests

AC#3 · rejects distance_threshold
// throws at the find_optimal_clusters boundary
await expect(
  find_optimal_clusters(data, {
    algorithm: 'agglomerative',
    algorithm_params: { distance_threshold: 2.5 },
  }),
).rejects.toThrow(/distance_threshold/);
AC#3 · accepts linkage & metric
// linkage + metric pass through and cluster
const res = await find_optimal_clusters(data, {
  algorithm: 'agglomerative',
  max_clusters: 4,
  algorithm_params: { linkage: 'average', metric: 'manhattan' },
});
expect(res.optimal.k).toBeGreaterThanOrEqual(2);
